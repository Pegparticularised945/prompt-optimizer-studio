# Prompt Quality Rubric

用于 `Review` 模式下的量化打分，保留当前线上已验证有价值的守卫：
**去重，不去骨；变薄，不变 mode。**

它要更稳定地区分：
- 是否保留原始任务目标与关键交付物
- 是否形成 `评分 -> 判定 -> 优化路径 -> 单一交付` 的闭环
- 是否避免把“看起来更完整、实际上更重更慢”的改写错判为提升
- 是否避免为了追求更短，直接把 `Review / Debug` 压扁成 `Create` 或业务执行器

## Scoring Dimensions

1. 目标清晰度（15）
2. 输入约束完整度（10）
3. 输出契约明确度（15）
4. 逻辑闭环（15）
5. 可执行性（10）
6. 鲁棒性（10）
7. 防幻觉与证据约束（10）
8. 反死胡同能力（10）
9. 可迭代性（5）

## High-Score Interpretation

高分提示词不只是“没错”，还必须体现 **必要且克制**：

- **目标清晰度**：任务锚点单一明确，不混入无关元任务，不把原任务包成更泛的说明文。
- **输入约束完整度**：覆盖真正必要的变量与缺参处理；不靠罗列大量低价值变量制造“严谨感”。
- **输出契约明确度**：输出结构足够清晰、可直接执行；若只是重复标题、重复守卫、重复交付规则，不应高分。
- **逻辑闭环**：评分、判定、优化路径与最终交付前后衔接；同一判断不应在多个章节机械重复。
- **可执行性**：拿到输入后能稳定落地；前置协议、元说明、章节密度不能压过真实任务本身。
- **鲁棒性**：有必要的异常处理、缺参处理与回退策略；但不能把所有边角都堆成冗长流程。
- **防幻觉与证据约束**：明确哪些内容必须锚定输入证据，哪些只能写“未提供 / 证据不足”。
- **反死胡同能力**：能识别卡死信号并给出切换条件；不是只堆风险词汇。
- **可迭代性**：后续容易继续收紧，而不是因为结构过厚导致每轮都只能加法膨胀。

## Hard Guards

### 1. 证据触发
- 只有当输出形态、主交付或核心约束真的被改坏时，才判负面；不能因为出现结构化输出词汇就自动扣分。

### 2. Mode-Fidelity Guard
精炼只能发生在 **当前任务模式内部**，不能通过改写任务模式来“变短”。

- 若原任务是 `Review`，高分必须仍然保留：评审对象、评分/判定、优化路径、单一最终交付这条主线；不能直接塌缩成业务执行 prompt。
- 若原任务是 `Debug`，高分必须仍然保留：失败定位、最小修复、回归检查；不能直接改成从零重写。
- 若原任务是 `Create`，则应直接交付目标 prompt，而不是额外套一层 Review 审计外壳。
- **精炼只能发生在当前任务模式内部**；为了更短而改变任务模式，属于漂移，不属于精炼。

### 3. 契约保留
- 遇到 `Debug` 或结构化任务时，优先保留原主交付与原业务契约，再补字段完整性、回退和失败处理。

### 4. 抗虚假完整度
- 章节更多、术语更密、守卫更厚，不等于更高分；没有新增可执行行为时，应视为冗余而非优势。

## Density & Necessity Guards

注意：**长度本身不是扣分项**。只有当长度没有换来新的可执行行为时，才视为负担。

- 若同一约束在多个段落反复出现，但没有新增行为差异，`输出契约明确度` 与 `可执行性` 应扣分。
- 若提示词把大量篇幅花在元说明、协议管理、重复守卫，而不是目标任务本身，不应进入最高分段。
- 若标题数量、硬性规则、步骤层级明显超过任务复杂度，属于“正确但过胖”，不能按精炼高质量提示词给高分。
- 若只是“章节更多、措辞更满”，但没有提高结果稳定性、抗漂移性或可执行性，不得加分。
- 若同一死胡同规则、单一交付规则、漂移守卫被重复书写 2 次以上且语义基本相同，通常不应高于 `94`。
- 若 prompt 已经能完成任务，但仍出现明显的协议臃肿、重复标题、重复契约，通常应落在 `85-94`，而不是 `95+`。

## Additional Review Guards

- 若优化结果更泛化、更空泛或更像方法论说明，不得判高分。
- 若出现多个并列候选版本，不得判高分。
- 若删除原任务关键产物、关键边界或关键业务词，必须判为严重漂移。
- 若优化结果虽然正确，但通过堆叠重复规则获得“完整感”，不得因为更长而加分。
- 若为追求更短而删除 `Review / Debug` 所必需的闭环结构，不得判高分。

## Decision Threshold

- `>= 85`：高质量，只做微调。
- `70-84`：中质量，保留结构并做定向修复。
- `< 70`：低质量，切换统一框架重构。

## Score Band Notes

- `95-100`：目标稳定、mode 稳定、契约清楚、执行负担低；几乎没有无效重复，结构与任务复杂度匹配。
- `90-94`：整体正确且可用，但能看出轻微冗余、协议偏厚或局部重复。
- `85-89`：核心方向正确，但存在明显的“过度规定 / 过度分段 / 过度重复”问题。
- `< 85`：除结构负担外，还存在模式漂移、目标漂移、契约断裂、执行不稳或关键守卫缺失等问题。
